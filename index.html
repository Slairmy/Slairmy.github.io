Hexo 搭建简约版博客

发表于 2020-01-01

2019年的最后一天了，Hexo搭建的新博客迟迟未通过Github Page部署。新的一年开始，决定要把它部署好！2020年又是一个新的开始！

Hexo搭建个人博客

通过官方文档也是很容易的，按照文档上面的操作，简单来说就分为几步，同时依赖nodejs的执行环境。

1、安装Hexo脚手架（全局安装）

npm install -g hexo-cli

2、使用Hexo构建项目

Hexo init project-name

3、安装所需包

cd project-name && npm install

一次执行完上述步骤就可以在project-name目录下看到文件结构了：

在项目根目录下执行Hexo server默认访问本地4000端口即可看到页面，本身是一个快速搭建博客的工具，博客站点的很多配置都是通过配置项来设置的，主要的配置项文件是_config.yml。关于配置项具体看文档介绍。

Github Page部署

个人在搭建后并且修改部分配置之后，按照文档中的部署步骤部署主要有如下的坑：按照Travis CI的部署步骤无法修改Github Page下面的发布分支！这里做下记录。

文档对于Github Page的部署有两种方式，一种是一键部署的简单方式，另一种是通过Travis CI(通过钩子自动检测主分支提交同步构建内容一样的gh-page分支)。这里我一开始使用的是后面那种部署方式：

首先创建一个 slairmy.github.io 的仓库，然后步骤如下：

1、将 Travis CI 添加到github的账户中

2、在 github 的 Applications settings 添加 Travis CI 权限

3、生成 Personal Access Token 并在 Travis CI 配置

推送到远程 master 分支之后，在 Travis Ci 中也显示构建成功了。但是访问确是README中的部分信息。而且在 Github Page 的设置信息中都没有显示其他的分支。

查阅一些资料，发现 <你的 GitHub 用户名>.github.io 这种命名方式是只能构建 master 分支。也就是说 <你的 GitHub 用户名>.github.io 命名仓库的 Github page 只能从 master 分支发布。其他的项目页面不以 <你的 GitHub 用户名>.github.io 可以选择其他的分支作为发布源。

所以最终选择了一键部署这种策略，一键部署需要安装一个npm包: 如下

npm install hexo-deployer-git --save
